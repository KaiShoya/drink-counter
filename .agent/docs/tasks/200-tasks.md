#200: `/drinks` を `/labels` の下へ移動し、ラベル毎の順番入れ替えを実装

最終更新: 2026-01-01

目的: ユーザーがラベル毎に飲み物の表示順を入れ替えられるようにし、ページ構造として `/labels/:labelId/drinks` のようなネストされたルートを提供する。

範囲:
- 既存の `/drinks` ページを `/labels` 配下に移動（ルーティング変更）。
- 各ラベルごとにドラッグで順序を入れ替え可能にするUIを実装。
- 順序を永続化するAPI/リポジトリ変更と、必要なDBマイグレーション。
- 既存の一覧/集計表示への影響確認（ホーム画面や集計画面でのソート挙動）。

要件（機能）:
1. ラベル詳細ページ（例: `/labels/:id`）から、そのラベルに属する飲み物を順序付きで一覧できる。
2. ユーザーはドラッグ&ドロップでラベル内の飲み物順を変更できる。
3. 変更は即時保存され、同一ユーザーの他セッションで反映される。
4. 飲み物が複数ラベルに属する場合、順序管理は「ラベルごと」に行われる（各ラベルに対して別個の順序が必要）。
5. 元のトップレベル `/drinks` はオプションで残すかリダイレクトする（要確認）。

非機能要件:
- モバイルでの操作性（ドラッグの代替手段）を確保する。
- 変更操作は最小限のAPIコールで済むようにバルク更新を検討する。

データモデル案:
- 本件については「飲み物は一つのラベルに紐付く（複数ラベルは許可しない）」という前提で進めます。
- 順序はユーザー毎に保持するため、`drink_labels` のような中間テーブルではなく、`drinks` テーブル側に `position` および `user_id` を組み合わせた管理を検討します。
- ただし、DB マイグレーション作業は依頼主が実施するとのことなので、マイグレーションファイルの作成・適用は今回の実装範囲外です（後述）。

API / リポジトリ変更:
- `drinkLabelsRepository` / `drinksRepository` に以下を追加/変更:
  - ラベルに紐づく飲み物を順序付きで取得する `fetchByLabel(labelId)`（position 昇順）。
  - 順序を更新する `updatePositions(labelId, [{ drinkId, position }, ...])` のようなバルクエンドポイント。

フロントエンド設計:
- ラベル詳細ページにドラッガブルリスト（`vuedraggable` 等）を導入。
- ドロップ完了時に位置配列を送信して永続化。
- `/drinks` を移動する際、既存コンポーネントは可能な限り再利用する（移行コスト低減）。

受け入れ基準:
- ラベル詳細でドラッグで順序を変更できる。
- 順序変更を行った後、ページ再読み込みや別セッションで変更が反映される。
- 既存の一覧機能（ホーム等）でソートの破壊がない。

実装タスク（高レベル）:
1. 仕様確定（このファイルの「確認事項」をクリア）
2. DBマイグレーションの作成（`drink_labels.position` 追加案）
3. リポジトリ/API の実装（バルク更新エンドポイント）
4. フロントエンド: ルート移動、UI の実装/既存コンポーネントの移行
5. テスト（ユニット・E2E）と `.agent/specs/` の更新
6. PR 作成・レビュー対応


決定事項（ユーザーの回答に基づく）:
1. 飲み物は**一つのラベルに紐付く**。複数ラベルは許可しない。
2. 順序は**ユーザー毎**に保持する。
3. トップレベルの `/drinks` は**移行**する（トップレベルは削除またはリダイレクトしてラベル配下へ統合）。
4. DB マイグレーションは依頼主が実施するため、こちらではマイグレーション適用作業を行わない（設計・API・フロント実装を提供）。

次のアクション提案:
- DB マイグレーション作業は依頼主が実施するため、こちらはマイグレーション作成を行わず、以下に集中します:
  1. リポジトリ / API 変更の設計・実装案（`fetchByLabel(userId,labelId)`, `updatePositions(userId,labelId, [{ drinkId, position }, ...])` など）
  2. フロントエンド側のルート移行（`/labels/:labelId/drinks`）とドラッグ&ドロップ UI 実装案
  3. 単体テスト・統合テストの追加案と `.agent/specs/` の更新案

残課題:
- UX改善
  - ラベル詳細表示と更新をボタンで切り替えるようにする
    - ラベル名タップで詳細、更新ボタンタップで更新モードで開く
  - ドラッグが失敗した際のリカバリ（UNDO トースト）を検討

- DB マイグレーション（依頼主担当）
  - `drinks.sort`（または `position`）と `drinks.user_id` の整合性を確保するマイグレーション
  - 既存データの初期 sort 値採番ルール（作成日時順 or 既存 sort を利用）を確定する
  - RLS ポリシーで `user_id = auth.uid()` を維持しつつ、label スコープの更新が可能な設計を確認する

- ラベル編集リセット問題
  - 症状: 飲み物編集後にラベル詳細に遷移したときに、編集フォームの状態がリセットされることがある
  - 対応案: ラベル詳細と編集を分けることで解消予定

- 既存の飲み物画面の削除
